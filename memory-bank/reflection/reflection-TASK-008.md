# Reflection: TASK-008 - Cc'd Contact Self-Verification

**Date**: 2026-05-09
**Task Complexity**: Level 3 (inherited from FEAT-006)
**Total Phases**: 3 active (+ 3 deferred to FE pass)
**Duration**: 2026-05-09 (single working day, plan → creative → 3 build phases)

## Executive Summary

TASK-008 delivered the backend foundation for CC'd contact self-verification: a schema migration dropping dead `display_name` and introducing three canonical identity columns, a `Contact#verified?` predicate derived from field presence, a 10-line US-E.164 phone normalizer PORO, and a gating filter in `OnboardingFlow::EscalationCascade.fallback_emails_for` that silently drops unverified contacts from escalation fanout while preserving legacy raw-string fallbacks. All 3 active phases committed cleanly; the suite grew from 382 to 414 specs with zero failures; `rubyfmt --check` exits 0 globally.

The most notable event of this task was an early empirical finding by the Spec Writer Agent: `display_name` — the column the entire task description was built around — had been dead schema since FEAT-001 with zero reads or writes anywhere in the app or test suite. This finding re-shaped Q3 from "how do we split an existing name field?" to "we are dropping a phantom column", a fundamentally different and simpler migration. The creative phase resolved this cleanly, but the root cause — an incorrect assumption in the task description carried forward from the roadmap — is a workflow gap worth addressing.

The user's mid-cycle scope cut (all FE work deferred after all three creative agents had written their docs) was handled without incident at the build layer. The Live-Dogfood-Pending Tracker captured the three deferred items cleanly. However, the scope cut required manual re-writing of the Implementation Roadmap and Test Strategy in the task file, revealing that those sections are brittle to post-creative scope changes — they should be regenerated by a lightweight sub-agent rather than edited manually.

## Dimension 1: Task Implementation Quality

### Requirements Achievement

**Status**: Partial (active phases fully met; FE phases intentionally deferred)

The three active backend phases all delivered exactly what the architecture doc specified:

- **Phase 1** (commit `af9c745`): Migration `RemoveDisplayNameFromContactsAndAddIdentityFields` drops `display_name` (with type hint for reversibility) and adds `first_name :string`, `last_name :string`, `phone :string`. `Contact#verified?` and `Contact#unverified?` predicates, `:verified`/`:unverified` scopes, `encrypts :phone` (non-deterministic), `nullify_blank_identity_fields` callback, and factory traits all land as specified.

- **Phase 2** (commit `9a01d5d`): `Contacts::PhoneNormalizer` is 10 lines of logic, correctly handles 10-digit, 11-digit-with-leading-1, formatted, already-E.164, and international rejection. Returns `nil` directly (not a `Result` struct — see deviation note below).

- **Phase 3** (commit `3af0344`): `EscalationCascade.fallback_emails_for` implements the three-rule semantics precisely: verified contacts kept, unverified dropped, unknown raw strings kept. The gm_nudge `fallback_chain` payload inherits the same filtered list, which is the TASK-007 interaction point explicitly called out in the build notes.

One structural deviation from the architecture doc: `Contacts::PhoneNormalizer` was implemented as a simple module function returning `nil`/`String` rather than the `Result = Struct.new(:normalized, :valid?, keyword_init: true)` value object prescribed in the architecture doc. The simpler return type is arguably cleaner for a pure normalizer with only two outcomes, but it diverges from the `_learned/service-shape.md` directive that services with more than one outcome should return a typed struct. The actual downstream consumers (the deferred `Contacts::VerificationsController`) are not yet written, so the mismatch has not caused a compatibility issue — but when the FE phase lands, the controller shape in the architecture doc assumes `phone_result.valid?`, not a nil check. This should be reconciled before the deferred FE phase begins.

The three acceptance criteria that are fully exercisable backend-only (AC-GATING-1 via the cascade spec, and the two factory-trait-driven model specs) all pass.

### Code Quality Assessment

**Overall Rating**: Good

- **Maintainability**: High. Each phase introduced one well-scoped unit. The migration is reversible. The normalizer is a standalone module with no dependencies. The cascade filter is a single private method change with explicit comment citing FEAT-006/TASK-008 and explaining the three-rule semantics. Comments are sparse but well-placed.

- **Architecture**: Good. `nullify_blank_identity_fields` as a `before_validation` callback is the right place for blank-to-nil coercion — it fires on all persistence paths without requiring callers to know about it. The `:verified`/`:unverified` scopes correctly model the derived-state logic from the predicate. The scope implementation for `:unverified` (using `.or(where(last_name: nil)).or(where(phone: nil))`) is correct but slightly verbose — an `.invert_where` or a complement-of-verified scope could have been used. Not a problem at this scale.

- **Error Handling**: The PhoneNormalizer returns `nil` for all invalid inputs cleanly. No exceptions are raised. The cascade filter guards against `nil` contact via `contact && !contact.verified?` rather than a nil check on `verified?`, which is the right defensive posture.

- **Testing**: 32 new specs across three files. Coverage is thorough for the backend surfaces shipped. The verified?/unverified? matrix tests are well-structured using the `:verified` factory trait with individual field overrides. The cascade gating tests cover all three rules explicitly (skip-unverified, pass-verified, pass-raw) and include the gm_nudge `fallback_chain` payload consistency test — a non-obvious cross-concern that was explicitly tracked in the task notes.

### Technical Decisions

**Key Decisions:**

1. **Q3-A: Drop `display_name`, add canonical identity columns** — The dead-schema finding made this a straightforward call. Rationale holds: cohabiting a never-written column with three new identity fields would create a permanent "which wins?" question. The migration is reversible (type hint on `remove_column`). Zero data lost because zero rows ever populated it.

2. **Q2-A: Suppress escalation fanout only (not submission prompts or digest)** — Scoped to the highest-leverage gate with the smallest surface. Self-healing on verification. The implementation correctly handles the empty-fallbacks-after-filter case by falling through to gm_nudge (the cascade already handles that path). This decision was the one place where the Architecture and User Journey agents had a real tension: the User Journey agent initially recommended Candidates C+D (non-blocking) because it was worried about a verification deadlock, but the Architecture agent chose Candidate A (hard gate) after reasoning that gm_nudge is the implicit escalation for the case where all fallbacks are unverified. The orchestrator reconciled this correctly.

3. **Q4-C: In-house US-E.164 normalizer + non-deterministic `encrypts :phone`** — The 10-line normalizer is exhaustively tested. Non-deterministic encryption is correct for a write-once field not currently queried. The `phonelib` deferral is explicitly documented with a clear migration path for when Twilio integration lands.

**Trade-offs:**

- **No `Result` struct from PhoneNormalizer**: Gained simplicity (nil vs string is idiomatic Ruby for "nothing or a value"); lost type-consistency with the rest of the service layer and created a divergence from the architecture doc's specified controller integration shape.

- **No model-level presence validations on identity fields**: Gained the ability for `Contact.find_or_create_for_email` to create unverified contacts without errors; requires the future controller to manage its own validation. This is the correct trade-off for the domain model but must be remembered when the FE phase arrives.

### What Went Well

1. **Architecture doc fidelity**: All three active phases landed precisely as the architecture doc specified. The three-rule cascade semantics (verified KEEP, unverified DROP, unknown raw passthrough) were implemented correctly on the first attempt with zero test failures post-write.

2. **Factory trait design**: The `:verified` factory trait using `sequence(:first_name)` rather than a static value is a subtle but good choice — it prevents name collisions in specs that create multiple verified contacts within the same example, which the cascade gating spec does directly (`"verified@smithtoyota.com"` and the fallback chain test both create `:verified` contacts).

3. **`nullify_blank_identity_fields` callback**: Proactively handling blank-to-nil conversion at the model level rather than expecting every caller to send `nil` explicitly is a good defensive pattern. It ensures `Contact#verified?` (which uses `.present?`) and the database scopes (which use `WHERE x IS NOT NULL`) agree on what "blank" means.

4. **gm_nudge payload consistency test**: The third cascade spec (`"filters the gm_nudge fallback_chain payload identically"`) is particularly valuable — it tests a cross-concern (TASK-007's `fallback_chain` payload) that would be easy to forget and is exactly the kind of integration regression that's painful to debug in production.

5. **Phase commit granularity**: Six commits on this branch (planning + creative in `6a12ddb`, three phase commits, one memory-bank update commit) make the branch easy to review and bisect.

### Challenges Encountered

1. **`display_name` was dead schema, task description was wrong** — The task description said "split `Contact#name`" but the column was `display_name`, never written or read. The Spec Writer Agent caught this via grep during planning and documented it as Q3 in the spec. The architecture creative phase then properly resolved it. Resolution: Q3-A drop-and-replace. Lesson: see Extractable Learnings.

2. **rubyfmt replacing rubocop in the lint gate** — The build commands in `techContext.md` (or the rai-build command's default) assumed `rubocop` as the linter. This project uses `rubyfmt --check` instead. This had to be discovered and corrected mid-task (the Execution State log notes "rubyfmt --check exits 0 globally" as a positive signal in each phase). This is infrastructure debt surfaced by the task rather than introduced by it.

3. **Post-creative scope cut required manual roadmap reconstruction** — After all three creative agents completed their docs, the user directed that all FE work be deferred. The Implementation Roadmap (which had been written to 6 phases matching the creative docs) and the Test Strategy (which targeted 12–16 tests) both had to be manually rewritten to reflect the 3-phase backend scope. The creative docs themselves were not wrong — they remain valid for the deferred FE pass — but the Roadmap and Test Strategy sections are downstream of both planning and creative, meaning any post-creative scope change currently requires manual reconstruction of those sections.

### Technical Debt & Future Work

- **`Contacts::PhoneNormalizer` return type diverges from architecture doc**: The architecture doc specifies `Result = Struct.new(:normalized, :valid?, keyword_init: true)` and the `VerificationsController#update` shape uses `phone_result.valid?`. The implementation returns `nil` or a String. Before the deferred FE phase, either the normalizer should be updated to return a `Result` struct (preferred for consistency with `_learned/service-shape.md`) or the architecture doc's controller snippet should be updated to use nil-check idiom. This is not a bug today but is a forward-compatibility risk.

- **Deferred FE phases (tracked in Live-Dogfood-Pending Tracker)**: `Setup::WalkthroughsController` identity step, `OnboardingMailer#invitee_setup_email` copy edits, and end-to-end system spec. Full UI/UX spec is in `memory-bank/creative/TASK-008-uiux.md` and remains valid — the deferred FE pass can begin directly from that document.

- **`Contact.verified` scope on encrypted `phone` column**: The architecture doc notes this scope will not use an index efficiently because `phone` is non-deterministically encrypted. At current scale (the `:verified` scope is never the entry point — gating reads happen per-contact via `verified?`), this is not a problem. If a feature ever needs to batch-query verified contacts at scale, the scope would need rethinking (e.g., adding a `verified_at` timestamp column as a queryable proxy).

## Key Learnings

### Extractable Learnings (for Continuous Learning)

De-duplicated against existing `_learned/` rules: existing rules cover audit-trail (FlowEvents), idempotency, namespacing, service-shape, time-zones. None of the following duplicates those.

1. **schema-validation** (`db/schema.rb`, task specification, `memory-bank/tasks/`): Before writing a task spec that names existing columns, grep those column names against `app/` and `spec/` to verify they are read or written somewhere — dead schema should be removed, not treated as a base to extend.

2. **service-shape** (`app/services/**/*.rb`): A service with exactly two outcomes (valid-result vs nil) may return nil directly rather than a Result struct, but document the choice explicitly and update any architecture doc controller stubs that assume `.valid?` before the consuming controller is written.

3. **scope-cut-resilience** (`memory-bank/tasks/`, build orchestration): When a user defers a feature scope after creative agents have written their docs, regenerate the Implementation Roadmap and Test Strategy sections from scratch against the reduced scope rather than editing them in-place — in-place edits leave dangling phase references and inconsistent test counts.

4. **gating-filter-passthrough** (`app/services/**/*.rb`, cross-model queries): When filtering a collection based on a related model's state, preserve items with no corresponding record as the default-keep case — only items whose record exists AND whose state fails the gate should be dropped; absence of a record is not a disqualification.

**Limits**: Level 3 tasks may extract 2–4 learnings. Four learnings extracted; all are genuinely reusable patterns that appeared concretely in this task.

### Learned Rules Applied

- **`_learned/audit-trail.md`**: Applied correctly. The architecture doc explicitly cites this rule for the `VerificationDispatcher` design (deferred FE phase).
- **`_learned/idempotency.md`**: Applied correctly in design. The architecture doc's `VerificationDispatcher` shape reads FlowEvent log before dispatch to prevent double-send. Not yet implemented (deferred phase), but the idempotency rule shaped the design.
- **`_learned/namespacing.md`**: Applied proactively. The PhoneNormalizer was placed under `Contacts::` (plural) per the rule, avoiding Zeitwerk collision with the `Contact` model class.
- **`_learned/service-shape.md`**: Partially applied. Architecture doc correctly specified a `Result` struct; implementation returned nil/String instead. Rule was loaded and cited but not followed in the final PORO implementation.
- **`_learned/time-zones.md`**: Loaded but not applicable to this task (no time-sensitive scheduling logic in the active backend phases).

### For Claude Code Workflow

1. **Spec Writer should validate schema assumptions** — Requiring the Spec Writer Agent to grep all column names from the task description against the actual codebase would prevent "split Contact#name" becoming three phases of work before someone notices the column is named `display_name` and was never written. The Spec Writer caught this through diligence, not through a systematic check.

2. **Creative-to-build transition needs a reconciliation gate** — The current workflow goes: plan → creative → build. There is no enforced step that re-reads all creative docs and validates the planning artifacts (Implementation Roadmap, Test Strategy) against them. For Level 3 tasks with 3+ creative agents whose outputs may contradict or supersede planning assumptions, a 5-minute reconciliation check would save 30 minutes of mid-build confusion.

3. **`techContext.md` development commands should be validated on each build** — The rubyfmt/rubocop mismatch is a canary for a broader problem: technical context documents drift. The rai-build command should sanity-check the documented lint and test commands before delegating to sub-agents. A failed lint command on step 1 is a poor first impression and slows the build unnecessarily.
