# Technology Context

## Technology Stack

### Runtime & Application Framework
- **Ruby**: 3.4.9 (managed via mise; `.ruby-version` pins this)
- **Rails**: 8.1.3 (full Rails — not API-only)
- **Bundler**: 2.x (lockfile committed; platforms include `arm64-darwin`, `x86_64-linux`, `aarch64-linux`)

### Database
- **PostgreSQL**: 18.3 (Postgres.app)
- **Adapter**: `pg` gem
- **Databases**: `rogue2_development`, `rogue2_test` (created); production schema mirrors `config/database.yml` with separate primary / cache / queue / cable databases per Rails 8 Solid* convention.

### Background Work, Cache, WebSockets (Solid stack — db-backed)
- **Solid Queue** — Active Job adapter for background jobs (`bin/jobs` runner). Persists to the `queue` database in production.
- **Solid Cache** — Rails.cache backend, persists to the `cache` database in production.
- **Solid Cable** — Action Cable backend, persists to the `cable` database in production.
- All three reduce infrastructure dependencies (no Redis required).

### Frontend
- **Hotwire**: Turbo + Stimulus, both via Importmap (no bundler step).
- **Asset Pipeline**: Propshaft (replaces Sprockets).
- **CSS**: stock Rails defaults at MVP — Tailwind not yet installed. Add when first UI feature ships.

### Inbound / Outbound Email (FEAT-001 dependency)
- **Action Mailbox**: not yet installed. **Required for FEAT-001** — provides inbound email routing (per-tenant onboarding addresses, per-(tenant, source) lead-ingest addresses). Install with `bin/rails action_mailbox:install` + `bin/rails action_mailbox:install:migrations` + `bin/rails db:migrate`. Production ingress TBD (Postmark / Mailgun / SendGrid / dedicated MX).
- **Action Mailer**: built-in. Used for outbound transactional email (single-click confirm, paced question emails, in-thread acknowledgments, setup-email magic links, weekly digest, escalation).
- **Inbound storage**: Action Mailbox stores raw RFC 822 source by default; raw payload archive (per the productBrief) extends this with indefinite S3-class retention. S3-class storage not yet wired up — Active Storage local disk at MVP development.

### Deployment
- **Kamal**: scaffolded (`config/deploy.yml`, `.kamal/secrets`, `.kamal/hooks`). Not yet configured for a real target.
- **Containerization**: Dockerfile generated by `rails new`, ready for Kamal-based deploy.
- **Web server**: Puma + Thruster (HTTP cache + sendfile acceleration).

### Observability
- Stock Rails `Rails.logger` with default request logging at MVP.
- OpenTelemetry / structured-JSON logging / distributed tracing per CLAUDE.md observability standards: **deferred until multi-service or production load** — single Rails monolith pre-launch does not warrant the tracing infrastructure overhead. Re-evaluate before first paid customer.

### Security & Quality Tooling
- **Rubocop**: `rubocop-rails-omakase` (Rails 8 default style).
- **Brakeman**: static security scanner, runs against the Rails app.
- **Bundler-audit**: vulnerability scanning of installed gems.
- **Web Console**: development-only, exception page debugging.

### Testing
- **Minitest** (Rails default) — `test/` directory present. Note: many RAI sub-agents and templates assume **RSpec** (see `${CLAUDE_PLUGIN_ROOT}/agents/build-test-*.md`). Decision deferred to first build phase: stay with Minitest (less ceremony, default tooling) or switch to RSpec (richer matchers, more idiomatic for the rai workflow). Capture decision in `systemPatterns.md` once made.
- **Capybara + Selenium WebDriver** for system tests (already in Gemfile `:test` group).
- **FactoryBot**: not yet added. Install when test factories become useful.

## Notable Project-Specific Considerations

- **Action Mailbox is a first-class platform capability**, not a bolt-on. The email-first onboarding flow (FEAT-001) means inbound email parsing — addressing scheme, threading, recipient parsing, signature stripping — is core to the product, and the MVP cannot ship without it.
- **Two distinct inbound email surfaces** must coexist:
  - Per-tenant onboarding-thread addresses (GM reply parsing)
  - Per-(tenant, source) lead-ingest addresses (ADF-XML and raw HTTP POST routing)
- **Indefinite raw payload retention** affects schema and storage decisions: every ActionMailbox::InboundEmail must be preserved indefinitely with full RFC 822 source, plus a parallel canonical record linked to the adapter version that produced it.
- **Single platform-wide encryption key at MVP** for PII fields (customer email, customer phone on leads). Use Rails 7+ `encrypts` with `deterministic: true` where lookup is needed.
- **Magic-link tokens** use Rails' built-in `signed_id` — short-lived, expirable, scoped per purpose.
